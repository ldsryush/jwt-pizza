# Incident: 2026-04-08 20-08-06

## Summary

Between **20:08 UTC and 20:37 UTC** on April 8, 2026, all users attempting to place pizza orders encountered HTTP 500 errors. The event was triggered by a chaos injection at the Pizza Factory at 20:08 UTC as part of a scheduled chaos testing exercise. The factory began returning failure responses instead of fulfilling orders.

The incident was detected by the Grafana OnCall `pizza_failures` alert rule shortly after. The on-call engineer resolved the incident at approximately 20:37 UTC by visiting the `followLinkToEndChaos` URL embedded in the error responses. This **high severity** incident affected 100% of users attempting to place pizza orders for approximately **19 minutes**.

No support tickets were submitted (test/academic environment).

## Detection

This incident was detected when the `pizza_failures` Grafana alert rule was triggered and the **JWT Pizza DevOps** on-call engineer (ldsryush) was paged via SMS through the Grafana IRM escalation chain.

The incident was also simultaneously detected by the local `scripts/monitor.ps1` script, which polled the order endpoint every 20 seconds and immediately surfaced the `followLinkToEndChaos` resolution URL in the terminal output at 20:08 UTC — before the Grafana alert's 1-minute pending period elapsed.

Reducing the alert pending period from `1m` to `None` will be implemented by the on-call engineer so that Grafana fires immediately on the first failure, cutting detection-to-notification time by up to 1 minute.

## Impact

For approximately **19 minutes** between **20:08 UTC and 20:27 UTC** on 04/08/2026, all pizza orders returned HTTP 500 errors with the message "Failed to fulfill order at factory."

This incident affected 100% of users attempting to place pizza orders. Orders were recorded in the database (order IDs were assigned) but no pizza JWT was returned, meaning no valid pizza was delivered to any customer.

0 support tickets were submitted (test/academic environment).

## Response

After the `pizza_failures` alert fired at approximately 20:09 UTC, the on-call engineer (ldsryush) came online and identified the `followLinkToEndChaos` URL displayed prominently in the terminal by the local monitor script.

The engineer visited the resolution URL in a browser at approximately 20:27 UTC, which disabled chaos injection at the Pizza Factory. No escalation to additional team members was required — the SMS notification and monitor script provided sufficient context for immediate resolution.

Response time from first failure (20:08) to resolution (20:27) was approximately **19 minutes**.

## Root cause

The Pizza Factory service injected chaos into the order fulfillment pipeline as part of a scheduled chaos testing exercise. The factory returned HTTP 500 error responses containing a `followLinkToEndChaos` field in the response body, pointing to a resolution endpoint. Visiting the resolution URL disabled the chaos injection and restored normal factory operation.

## Resolution

The incident was resolved by visiting the `followLinkToEndChaos` URL embedded in each factory error response:

```
https://pizza-factory.cs329.click/api/support/fb3ff04b8f9d45e3b7ea2fef33d5b457/report/e1f5p9tc
```

This URL was surfaced automatically by the `scripts/monitor.ps1` monitoring script, which parsed the JSON error response and displayed the resolution URL prominently in the terminal. After visiting the URL, orders immediately resumed (order #1038 at 20:27:35 UTC confirmed recovery).

Response time could be improved by reducing the Grafana alert pending period from `1m` to `None` and by logging the `followLinkToEndChaos` URL to Grafana Loki so it is accessible remotely without a local monitor script.

## Prevention

No prior incidents share this same root cause in this system's history.

Going forward, the same class of factory-level chaos injection could reoccur. To reduce future impact:

1. Log `followLinkToEndChaos` URL at `WARN` level in the service logger so the resolution URL is immediately visible in Grafana Loki during an incident — eliminating the dependency on a local monitor script.
2. Reduce the `pizza_failures` alert pending period from `1m` to `None` so Grafana fires on the very first failure.
