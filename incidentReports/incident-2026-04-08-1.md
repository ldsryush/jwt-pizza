# Incident: 2026-04-08 20-08-06

## Summary

Between **20:08 UTC and 20:37 UTC** (14:08–14:37 MDT) on April 8, 2026, all users attempting to place pizza orders encountered HTTP 500 errors. The event was triggered by a chaos injection at the Pizza Factory at 20:08 UTC as part of a scheduled chaos testing exercise. The factory began returning failure responses instead of fulfilling order.

The incident was detected by the Grafana OnCall `pizza_failures` alert rule, which triggered an SMS notification to the on-call engineer. The engineer investigated the failure by examining Grafana Loki logs, which surfaced the factory's error response body containing the `followLinkToEndChaos` resolution URL. The engineer resolved the incident at approximately 20:37 UTC by visiting that URL, which called the factory's support API and disabled chaos injection. This **high severity** incident affected 100% of users attempting to place pizza orders for approximately **29 minutes**.

## Detection

This incident was detected when the `pizza_failures` Grafana alert rule was triggered and the **JWT Pizza DevOps** on-call engineer (Sanghwa) was paged via SMS through the Grafana IRM escalation chain at approximately 20:09 UTC, roughly 1 minute after the first failure occurred.

Reducing the alert pending period from `1m` to `None` will be implemented by the on-call engineer so that Grafana fires immediately on the first failure.

## Impact

For approximately **29 minutes** between **20:08 UTC and 20:37 UTC** on 04/08/2026, all pizza orders returned HTTP 500 errors with the message "Failed to fulfill order at factory."

This incident affected 100% of users attempting to place pizza orders. Orders were recorded in the databa but no pizza JWT was returned, meaning no valid pizza was delivered to any customer.

## Response

After receiving an SMS alert from Grafana OnCall at approximately 20:09 UTC, the on-call engineer (Sanghwa) opened the Grafana dashboard and began investigating the spike in `pizza_failures` metrics.

The engineer navigated to **Grafana Loki** and searched the jwt-pizza-service logs for errors related to the order endpoint. The logs revealed HTTP 500 error responses from the Pizza Factory, including the full JSON error body:

```json
{
  "message": "Failed to fulfill order at factory",
  "followLinkToEndChaos": "https://pizza-factory.cs329.click/api/support/fb3ff04b8f9d45e3b7ea2fef33d5b457/report/e1f5p9tc"
}
```

The engineer visited the `followLinkToEndChaos` URL at approximately 20:37 UTC, which called the Pizza Factory's support API and disabled the chaos injection. Orders immediately resumed. No escalation to additional team members was required.

Response time from first failure (20:08) to resolution (20:37) was approximately **29 minutes**.

## Root cause

The Pizza Factory service (`pizza-factory.cs329.click`) injected chaos into the order fulfillment pipeline as part of a scheduled chaos testing exercise. This is an **external dependency failure**, the JWT Pizza Service itself was still working, but it depends on the Pizza Factory to sign and return pizza JWTs for every order. When the factory began rejecting all requests, no orders could be fulfilled.

## Resolution

The incident was resolved by examining the Grafana Loki logs for the jwt-pizza-service, which contained the full factory error response body including the `followLinkToEndChaos` URL. The engineer visited the URL in a browser:

```
https://pizza-factory.cs329.click/api/support/fb3ff04b8f9d45e3b7ea2fef33d5b457/report/e1f5p9tc
```

This called the Pizza Factory's support API, which disabled the chaos injection and restored normal order fulfillment. After visiting the URL, the next pizza order (order #1038 at 20:37:35 UTC) succeeded, confirming resolution.

Response time could be improved by ensuring the `followLinkToEndChaos` URL is explicitly logged at `WARN` level in the service logger whenever the factory returns it, making it immediately visible and searchable in Grafana Loki without needing to parse raw error body text.

## Prevention

1. Update `orderRouter.js` to explicitly log the `followLinkToEndChaos` URL at `WARN` level whenever the factory returns it, so the resolution URL is immediately searchable in Grafana Loki.
2. Reduce the `pizza_failures` alert pending period from `1m` to `None` so Grafana fires on the very first failure.

## Action items

1. Update `orderRouter.js` to log a `WARN`-level message containing the `followLinkToEndChaos` URL whenever it is present in a factory error response, making it visible and searchable in Grafana Loki.
2. Edit the `pizza_failures` Grafana alert rule: change pending period from `1m` to `None`.
3. Create a one-page runbook: "If pizza orders fail → check Grafana Loki for `followLinkToEndChaos` field → visit URL → confirm orders resume → resolve OnCall alert."
